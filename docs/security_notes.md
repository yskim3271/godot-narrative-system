# 보안 노트

## 1. 대화 데이터는 코드를 실행할 수 없습니다 (설계 보장)

조건/액션/시퀀서 명령은 GDScript `eval`도, Godot `Expression` 클래스도 사용하지 않습니다. 자체 파서가 해석하는 **닫힌 미니 언어**이며, 게임 코드에 닿는 길은 두 가지뿐입니다:
- `register_function()`으로 게임이 **명시적으로 화이트리스트에 올린 함수**
- 시퀀서 `call_method(actor, ...)` — 게임이 **명시적으로 레지스트리에 등록한 액터 노드**의 메서드만

즉, 신뢰 경계는 "무엇을 등록했는가"입니다. 등록 함수/액터에는 파괴적 동작(파일 삭제, 네트워크 등)을 노출하지 마세요.

## 2. `.tres` 파일은 코드입니다 — 신뢰되는 소스만

Godot의 리소스 파일은 **임의 스크립트를 임베드/참조할 수 있고, 로드하는 순간 그 코드가 실행될 수 있습니다.** 이는 Godot 전반의 특성이며 이 애드온의 데이터베이스(.tres)도 예외가 아닙니다.

- 인터넷에서 받은 대화 데이터베이스 .tres를 검토 없이 프로젝트에 넣고 열지/로드하지 마세요
- 유저 제작 콘텐츠(모드)를 받을 계획이라면 .tres가 아니라 **검증 가능한 중간 포맷**(JSON/CSV → 코드에서 리소스 구성)을 쓰세요 — `examples/integrated_demo/db_builder.gd`가 코드 구성 패턴의 예시입니다

## 3. 저장 파일은 안전합니다

- 저장/로드는 **순수 JSON** 경로만 사용합니다: `JSON.parse_string` → 타입 검사 → 상태 복원. `str_to_var`, `ResourceLoader`, 스크립트 인스턴스화가 일절 없으므로 **조작된 저장 파일로 코드를 실행시킬 수 없습니다.**
- 조작된 값은 방어적으로 처리됩니다: 타입이 틀린 섹션은 기본값, objective 카운트는 DB 기준 재클램프, 파싱 불가 파일은 격리 후 기존 상태 유지 (테스트로 고정: `test_save_hardening.gd`)

## 4. 시퀀서 리소스 로딩 제한

`play_audio`의 스트림 경로는 `res://` 접두사만 허용합니다 — 저장된 데이터/유저 입력이 디스크 임의 경로를 로드하게 만들 수 없습니다.

## 5. 기타

- id 문자셋(`[a-zA-Z0-9_.]`)을 벗어난 id는 검증기가 경고합니다 — 키/저장 데이터에 들어가는 값의 예측 가능성을 위한 것
- CSV 임포트는 데이터(문자열)만 만듭니다 — 수식/스크립트 해석 없음
